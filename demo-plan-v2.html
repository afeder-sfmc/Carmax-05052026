Phase 3 ⭐

Upper-Funnel Nurturing — Audience & Segment Creation

⏱ 25 minutes · CENTERPIECE 0:30 – 0:55
Pain: Strategists → Snowflake → data team → days Goal: Done in Salesforce, in minutes No SQL · No engineering ticket · No Snowflake login

📋 The Use Case — Sean's Own Words

"The big one we're working on now, which is a huge value driver, is upper-funnel nurturing. It's people that have done X, Y, and Z but haven't progressed. We could even do [things like] those things and they've clicked on the same car five days in a row — we would have that activity in Snowflake. Showing that capability would be a huge win."
— Sean Goetz, April 10, 2026

Segment logic (from the call):

✓ Downloaded the CarMax mobile app
✓ Favorited a car
✓ Clicked on the same car 5+ days in a row (strong buying signal)
✗ Has not submitted a financing application

Why it matters:

▸ This person is clearly interested — repeated engagement with the same vehicle is a high-intent signal

▸ They have not converted — the financing step is the friction point

▸ A timely, personalized email (referencing the specific car they've been looking at) can move them forward

▸ Currently: this segment requires Snowflake queries + data team handoff + wait time. In MCA: built by a strategist in under 10 minutes.

1

⭐ Open Segment Builder — Set the Persona (2 min)

Navigate to Data Cloud → Segments → New Segment. Emphasize who is doing this: a marketing strategist. Not a data engineer. Not someone with a Snowflake login.

Opening script: "I'm going to play the role of one of your strategists — someone who is trying to build a campaign audience and today has to go ask the data team to run queries in Snowflake. Watch how long this takes."

Demo env note: Select Unified Individual as the primary object. This is the identity-resolved profile that combines activity data from all sources — Snowflake events, app engagement, CRM data.
2

⭐ Build Filter 1 — Mobile App Download (3 min)

Add condition: Mobile App Download Event exists. Show the attribute picker — it surfaces all available data fields from the connected data sources. No SQL. No schema knowledge required.

What to click:

Add filter → Related Objects → Mobile App Events → Downloaded App → exists → Timeframe: All time

Talking point:

"This is activity data that came from Snowflake — your app event data. The strategist is selecting it in plain English, not writing a query."

3

⭐ Build Filter 2 — Favorited a Car (2 min)

Add condition: Favorites Event exists (vehicle favorited). Count: at least 1. Show count updating in real time as each filter is applied.

Pause and note the count updating. "Your strategist can see in real time how many people match as they add filters. No waiting for a query to run."

4

⭐⭐ Build Filter 3 — Same Car, 5 Days in a Row (5 min)

This is the high-intent signal Sean specifically described. It's the most complex filter — show it carefully.

⭐ The Money Moment — Linger Here

Add condition: Car Detail Page View — same Vehicle ID — Count ≥ 5 — within last 14 days. This is the filter that currently requires a Snowflake analyst to write a window function. In MCA Segment Builder, it's a filter configuration.

If the data schema allows: show the Vehicle ID grouping — "same vehicle" means the system is grouping page views by Vehicle ID and counting distinct days. This demonstrates the sophistication of the underlying data model.

Script: "This is the signal Sean described — someone who has looked at the same car five days in a row. In Snowflake, this is a window function. In here, it's a dropdown. Same result. Different world."

5

⭐ Build Filter 4 — Has NOT Submitted Financing (3 min)

Add exclusion condition: Financing Application Submitted does NOT exist (or = false). This is the "not yet converted" gate that makes this an upper-funnel audience vs. an active buyer.

Exclusion condition:

Financing Submitted Event → does not exist → All time
OR: Financing Application Status ≠ Submitted

Talking point:

"We're not emailing people who already applied — that would be noise. We want the people who are clearly interested but haven't taken the step yet."

6

Optional Add-on — Second Condition Group (3 min)

Add a second condition group with OR logic: also include people who started but abandoned a financing form in the last 30 days. Shows the AND/OR logic builder is accessible without SQL.

Talking point: "Same concept — abandoned a financing form is also a strong signal. Your strategist can add that as a second audience group with an OR — no engineering ticket, no new Snowflake query. It's just another filter."

7

Save the Segment — Show Count & Name It (1 min)

Name it: SEG_UpperFunnelNurturing_HighIntent_v1. Save. Show the final member count. Pause.

🎯 The "Before and After" Moment — Say This:

"That's the upper-funnel nurturing segment Sean described on our last call. Here's how long that just took." [pause — check the clock]. "In Snowflake, your strategist would spend hours on that query, then wait for the data team to pull it, clean it, and load it into ExactTarget. What you just watched took about ten minutes — and the person doing it never left Salesforce."

8

Activate — Connect Segment to a Flow (5 min)

Activate the segment. Create or show a pre-built Segment-Triggered Flow: when a contact enters this segment → wait 30 minutes → send the email built in Module 2 (with the personalized vehicle reference). Show the flow canvas.

Flow structure to show:

  • ▸ Trigger: Segment Entry (Upper-Funnel Nurturing)
  • ▸ Wait: 30 min (configurable)
  • ▸ Send Email: Upper-Funnel Nurturing Email v1
  • ▸ Decision: Opened? → Branch A (clicked CTA) / Branch B (no engagement)

Talking point:

"This is what replaces Journey Builder for this use case. It's Salesforce Flow — the same tool your core platform uses for any automation. Once the segment is defined, wiring it to a send is a few clicks."

🙋 Anticipated Questions — Have Answers Ready

Q:

"Our data isn't all in Salesforce — a lot of it is still in Snowflake or other sources. Does this still work?"

A: "Yes — Data Cloud ingests your Snowflake activity data via a data stream. Once it's mapped, your strategists access it through this same interface. The data stays in Snowflake as the source of truth. Your strategists just never need to touch it."

Q:

"What about the favorites and saved searches databases that aren't in Snowflake yet?"

A: "Those come in as additional data streams when your architects are ready. This is incremental — you don't have to connect everything at once. You can start with Snowflake activity data and add the others over time."

Q:

"How does this handle the 'same car' logic — that feels like it needs SQL?"

A: "For standard aggregations like that, Segment Builder handles it via the data model — Vehicle ID is a field on the Page View event, so 'same vehicle' is just filtering on that field. For more complex aggregations, Calculated Insights let a data person define the metric once in SQL, and then it shows up as a simple filter for the strategist."

Q:

"How often does the segment refresh? Is it real-time?"

A: "Segment membership updates continuously as data changes — when a new page view comes in or a financing application is submitted, the segment adjusts. The email trigger fires in near-real-time. No manual re-pull required."

Phase 4

Engineering Environment — Source Control, DevOps & Monitoring

⏱ 15 minutes 0:55 – 1:10
Note from Sean: This is the lowest priority for business partners but highest excitement for the engineering team. Lead with the business value first (faster releases, fewer production incidents), then go technical for Christine/Lawrence.
1

Business Lead-In (1 min) — Say This First

Script: "ExactTarget has no source control. That means your engineers can't review each other's work, can't roll back a bad deploy, and can't test changes before they hit production. Every campaign change is a production change. That ends here."

2

Source Control & Salesforce DX (5 min)

Show that MCA assets — email templates, flows, data graph configurations — are Salesforce metadata. Deployable via SFDX. Trackable in Git. PR-reviewable. Show a sample deployment command or the metadata structure. Sandbox → staging → production promotion via datakit or SFDX.

Key line: "Everything your team builds in MCA is versionable, deployable, and reviewable in a pull request — the same workflow your engineers use for everything else on the Salesforce platform."

3

Datakit Cross-Environment Deployments (4 min)

Show a datakit: packages up Data Cloud + MCA configuration — segments, flows, data graphs, email assets — and deploys it to another environment. This is how you promote work from sandbox to production.

Be transparent: some features require manual steps during deployment (flag which ones if asked). Don't oversell.

4

Monitoring & Observability (5 min)

Show flow execution tracking via the Flow DMOs (Flow Version Occurrence, Flow Element Run). Every send is logged and queryable. Show the email engagement DMO: opens, clicks, bounces in real time. Built-in dashboards — no external BI tool required for marketing ops monitoring.

Closing line: "Your engineers can debug a flow mid-execution. They can query who received what, when, and what they did with it. No more black-box sends."

Phase 5

Triggered & Event-Based Sends

⏱ 10 minutes 1:10 – 1:20
Context from Sean: Event-based sends are currently handled by an external system listening to the ESB, enriching the event, and pushing to Marketing Cloud. He wants a more native "sensing" capability. His word: "sensing."
1

Automation Event Triggered Flow (AETF) — 5 min

Show an AETF configured on a data event: when a new record matching "abandoned browse" or "high-engagement, no financing submission" appears in the Engagement DMO, the flow fires immediately. This is the native sensing Sean described — no external listener required.

Script: "This is what your ESB listener is doing today — watching for events and triggering actions. In MCA, that sensing is built into the platform. The event comes in, Data Cloud picks it up, the flow fires. No external middleware required."

2

Migration Path — Keep the ESB (3 min)

Reassure the room: CarMax doesn't have to rearchitect their ESB to use this. The existing ESB can continue detecting and enriching events — it just POSTs to the Data Cloud Ingestion API instead of the MCE API. Minimal change for the engineering team.

Key line: "Your ESB keeps doing what it does. The only thing that changes is where it sends the event at the end — Data Cloud instead of ExactTarget."

3

Progression Capability Service — Future State (2 min)

Briefly acknowledge that the Progression Capability Service (mid-funnel, individual next-step logic) is individual-level today, not queryable at segment scale. When CarMax integrates it via the Ingestion API, it unlocks both real-time triggered sends AND segment-level queries — without duplicating the real-time business logic.
