CarMax Today vs. Tomorrow
5 min
Goal: Anchor the audience in their own architecture so they see MCA as a natural evolution, not a rip-and-replace.
0:05 – 0:08

Current State (Their Words)

Reference the architecture: data lives in Snowflake/EDL → strategists query → data team handoff → ExactTarget send. Engineering bottleneck at every layer.

0:08 – 0:10

MCA in the Architecture

Show where Data Cloud + MCA slots into their Salesforce org. Snowflake remains the source of truth — Data Cloud ingests it via a data stream. Salesforce becomes the activation layer.

Key message: Snowflake doesn't go away. Strategists just stop having to touch it. The data source is transparent — they describe who they want, not how to query for them.
Architecture Call-outs
  • CarMax's Retail Org: Salesforce CRM (Sales + Service) already live — MCA sits natively on the same platform
  • Current Marketing: Journey Mgmt + Preferences handled via MC Connector → ExactTarget. MCA replaces the MCE layer, not the Salesforce CRM
  • Snowflake/EDL + Enterprise Data Warehouse → Data Cloud via Data Streams (no data duplication required with BYOL federation option)
  • Progression Capability Service, favorites/saved searches DBs: can be connected via Ingestion API or Data Federation — no full copy required
  • No migration of legacy content required — start fresh, run in parallel

20 min
Pain: Engineers required for email builds Goal: Designer/PM can build independently
Demo persona: A product manager or designer — no engineering access. They should be able to drag, drop, and publish. Reference Burke's personal CarMax email as the target fidelity.

Build an Email From Scratch

Create a new HTML-coded email using the visual editor. Drop in: header block (CarMax navy, logo), hero text, vehicle card tile (matching the component block Sean showed), CTA button. Show component blocks — designers can drop and edit, no HTML required.

0:20 – 0:25

Personalization with Handlebars

Show a simple merge field: Hi {{root.$dataGraph.FirstName__c}}. Add vehicle owned: "How's your 2019 Jeep Cherokee?" (using data from Data Graph). No AMPScript needed.

0:25 – 0:30

Preview & Send Test

Preview email with live data (render with a test record from the Individual DMO). Show mobile preview. Send a test to attendees in the room — they see a real email arrive.

Talking Points
  • Component blocks are reusable — brand team defines the blocks once; PMs and designers assemble them. Engineers are no longer in the critical path for every campaign
  • Images live in Salesforce CMS — same content repository as web/app; one source of truth for brand assets
  • Handlebars JS replaces AMPScript: {{root.$dataGraph.field}} — accessible to anyone who can write basic HTML
  • Personalization = using data already in the platform (name, vehicle owned, last browsed). Vehicle recommendations from external service still callable via Apex Class data source provider
  • WYSIWYG + code view available — designers use drag-drop, engineers can still access HTML directly

0:30 – 0:55
Module 2 — Audience & Segment Creation
25 min
Pain: Snowflake queries + data team handoff = days Goal: Strategist builds segment in minutes, in Salesforce Use case: Upper-Funnel Nurturing
Primary use case (from Sean): Upper-funnel nurturing — people who downloaded the mobile app AND favorited a car AND clicked the same car 5 days in a row, but have NOT progressed to financing submission. This segment currently requires strategists to go into Snowflake, write queries, partner with data teams, and wait "a long long long minutes."
0:30 – 0:35

Data Model Overview (2 min)

Quick orientation: Snowflake activity data → Data Stream → Data Lake Object → DMO (Individual, Contact Point Email, Engagement events). Identity Resolution creates Unified Individual. This is the profile strategists will query against.

0:35 – 0:42

Build the Upper-Funnel Nurturing Segment

Open Segment Builder. Select Unified Individual as base. Add filters: Mobile App Download = true, Favorited Car = true, Car Detail Page views ≥ 5 (same vehicle, last 7 days), Financing Submitted = false. Show record count updating live.

0:42 – 0:47

Segment Complexity — No Engineering

Add a second condition group: also include contacts who abandoned a financing form in the last 30 days. Show AND/OR logic builder. Demonstrate how a strategist can do this without writing a single line of SQL.

0:47 – 0:52

Activate the Segment → Flow

Save the segment. Activate it: show a Segment-Triggered Flow that fires when a contact enters the segment. Flow sends the email built in Module 1. Show the flow canvas — not Journey Builder, but comparable logic in Salesforce Flow.

0:52 – 0:55

Data Transparency Discussion

Address the Snowflake question directly: "Snowflake is still the source. What changed is your strategists never have to see it." Show the Data Stream config that ingests from Snowflake — one setup, ongoing refresh, transparent to the business user.

Talking Points
  • Segment Builder requires zero SQL — strategists describe the audience in business terms and Data Cloud translates it
  • Segments run against Unified Individual profiles — resolves identity across Snowflake, web, app, and CRM data sources
  • Real-time refresh: segment membership updates as data changes — no manual re-pull
  • Snowflake doesn't move. Data Cloud reads it (via Data Stream or Zero Copy/BYOL federation). The strategic insight stays in Snowflake; the activation happens in Salesforce
  • Favorites/saved searches DB and preferences DB: can be added as additional data streams when CarMax's architects have them ready — incremental adoption, not big bang
  • Calculated Insights (SQL-based CIs) available for power users who want to define engagement scores or complex aggregates — still no engineering ticket required
Prep needed: Sean to share 1–2 real segment examples (the lady who builds these is out — follow up Monday). Need sample activity data in the demo org that mirrors CarMax's Snowflake schema: mobile app events, car detail page views, financing form events.

Datakit Deployments

Show datakits: package and deploy Data Cloud + MCA configuration across environments. One command to promote a campaign from sandbox to production. Flag: some manual steps still required for certain features — be transparent.

1:05 – 1:10

Monitoring & Observability

Show Flow execution tracking: Flow Version Occurrence, Flow Element Run DMOs — every send is logged and queryable. Show email engagement reporting: opens, clicks, bounces in real-time via Email Engagement DMO. Built-in dashboards.

Talking Points
  • MCA is metadata-first — everything is versionable, deployable, reviewable in a PR workflow
  • CI/CD with GitHub Actions or any standard Salesforce DevOps pipeline (Copado, Gearset, etc.) — fits existing enterprise DevOps practices
  • Debug flows mid-execution: Flow debug logs available in the platform — no more black-box sends
  • Monitoring: Data Cloud flow execution DMOs create an audit trail for every send event — critical for compliance and incident investigation
  • AI capabilities available for engineers: LowCode tools, Agentforce integration, agentic automation — speak to the "AI" capability icon on their architecture diagram

1:10 – 1:20
Bonus — Triggered & Event-Based Sends
10 min
Current: External ESB listener enriches events → MCE Goal: Native "sensing" in Salesforce platform
Sean confirmed: Triggered sends (abandoned experience, event-based) are currently handled by an external system listening to the Enterprise Service Bus, enriching the event, and kicking it to Marketing Cloud. He wants a more native "sensing" capability.
1:10 – 1:15

Automation Event Triggered Flow (AETF)

Show an AETF configured on a data event: when a new record appears in the Engagement DMO matching "abandoned financing form," the flow triggers immediately. No external ESB listener required. Show the trigger config and the flow canvas.

1:15 – 1:20

Ingestion API for Real-Time Events

Briefly show: CarMax's Progression Capability Service or ESB can POST events directly to Data Cloud via the Ingestion API. The event lands in a DLO → triggers a flow. The external enrichment logic stays external; only the activation moves into Salesforce.

Talking Points
  • AETFs fire in near-real-time as data events occur in Data Cloud — this is the "sensing" Sean described
  • CarMax's existing ESB can continue to own event detection and enrichment; just change the final destination from MCE API to DC Ingestion API — minimal rearchitecture
  • Progression Capability Service is individual-level, not queryable at segment scale today — Ingestion API integration would unlock both real-time triggers AND segment-level queries over time
  • This approach avoids duplicating real-time business logic that already lives in CarMax's systems
